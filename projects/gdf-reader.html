Summary

This is a small file parser made for reading key-value pairs similar to that of a TOML file. This was initially created for storing game data while creating text based games, but it can also be used for any other type of data storage. This parser, and schema, is not particularly robust specifically when it comes to storing strings, but it's main features is its compatibility with parsing out enums, which can be achieved by making the enum types implement the GDFCompatible interface.

Examples

Data Types

<data>
:x = 1
:y = 5
:str = Hello_World!
:allow = +
:ele = fire

This example demonstrates how data is stored within the gdf file. The data tag, contains 5 fields: x, y, str, allow, and ele. Traditionally, fields x and y would contain an integer and str, allow, and ele contains a string. Of course, by specification, allow should be accessed as a boolean with '+' and '-' being respective analogs for 'true' and 'false.' And, out of convenience, ele can be interpreted as an enum value. In this case, the possible values may be earth, water, fire, or air.

Due to the simplicity of the parser, there is no way of determining the literal value of each field. The x field could be interpreted as either a string, integer, or floating point number. The type of data read out of the file is up to the calling code.

Player Classes

<knight>
:health = 100
:strength = 10
:attack_type = melee

<wizard>
:health = 80
:strength = 5
:attack_type = magic

This example demonstrates how this data might be used within a game. Each character class has a default health and strength field and an enum attack_type.
